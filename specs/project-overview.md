## Tech stack (target)

The codebase currently ships with **Next.js** (App Router), **React**, **TypeScript**, and **Tailwind CSS**. The spec below assumes these additions as the app grows:

- **Database**: PostgreSQL on **Neon**, accessed with **Prisma** (server-only).
- **Auth**: **better-auth** (sessions, email/password; roles for customer vs admin).

## What does this app do?

Pixel Vault is an online store where users can browse, search, and filter video games, save favorites to their account, and admins can manage the product catalog.

## Users

### Customer

A gamer aged sixteen to fifty browsing on a laptop or phone. They know what they want: search, discovery, browse, and filter. They want to save games for later, not necessarily buy right now. They care about speed and clean design; they do not read instructions.

### Admin

The store owner, not technical, who wants to add a new game without calling a developer. They need a simple form (title, slug or auto-slug, price, genre, platform, image URL) and a way to remove a product that is discontinued.

## Scope (MVP)

- **No cart, checkout, or payments in v1.** The game detail page may show a **Buy** control as a placeholder (disabled, “coming soon,” or link-out to an external retailer—pick one implementation and keep it consistent).
- Core flows: catalog discovery, favorites, admin CRUD for products, sign-in/sign-up.

## Routes

### Public (no login)

- **`/`** — Homepage: featured games hero + full catalog.
- **`/games/[slug]`** — Game detail: full info, MVP buy placeholder (see [Scope](#scope-mvp)), favorite button (prompts login if not authenticated).

### Authenticated (must be logged in)

- **`/favorites`** — Saved games.
- **`/auth/sign-in`** — Sign-in form.
- **`/auth/sign-up`** — Sign-up form.

### Admin only (logged in + `ADMIN` role)

- **`/admin`** — Dashboard: product list with add/delete.
- **`/admin/products/new`** — Create product form.

## Architecture rules

- Default to **Server Components**.
- Add `"use client"` only for `useState`, `useEffect`, event handlers, or browser APIs.
- Never use `useEffect` for data fetching — fetch in Server Components directly.
- Prefer **Server Actions** for app-owned form mutations.
- Prefer **Route Handlers** for public APIs, webhooks, or third-party integrations.
- Keep server-only code out of Client Components.
- Shared UI lives under `components/`.
- Shared logic lives under `lib/` or `hooks/`.
- If Prisma is present, use it only in server code.

## Page behaviors

### Homepage (`/`)

**What they see**

- Hero with featured games (`isFeatured: true`).
- Filter bar: genre buttons + platform buttons.
- Search input.
- Product grid — all games, filtered by active selections.

**What they can do**

- Search by title → grid updates.
- Click genre / platform → grid filters.
- Click a game card → navigate to `/games/[slug]`.

**Implementation notes**

- Interactive search + filters → Client Component (`ProductFilters`).
- Initial catalog fetch → Server Component; pass data into the client filter UI.
- Data flow: Server Component fetches → passes props to Client Component → filtering in the browser.

### Game detail (`/games/[slug]`)

**What they see**

- Large game image.
- Title, genre, platform, publisher, release year.
- Price and description.
- “Add to Favorites” or “Favorited ✓” when logged in; “Sign in to save” when not.

**What they can do**

- Add favorite → Server Action.
- Remove favorite → Server Action.

**Implementation notes**

- Favorite toggle → Client Component (`FavoriteButton`) with `initialIsFavorited: boolean` from the Server Component.
- Page loads game by slug and checks favorite state → Server Component.

### Favorites (`/favorites`)

**Protected** — redirect to `/auth/sign-in` if not logged in.

**What they see:** grid of favorited games.  
**What they can do:** open a game, remove from favorites.

Server Component reads from the database directly. No list filtering beyond “my games.”

### Admin (`/admin`)

**Admin only** — redirect if not admin.

**What they see:** table of products, delete per row.  
**What they can do:** “Add Product” → `/admin/products/new`; “Delete” → Server Action (with confirmation).

## Auth pages

**Redirect if already logged in** — no empty login UX for signed-in users.

### Sign in (`/auth/sign-in`)

**What they see**

- Centered form card.
- Email, password (show/hide), “Sign In,” errors on bad credentials, link to sign up.

**What they can do**

- Submit credentials, toggle password visibility, go to sign up.

**On submit**

- Calls better-auth `signIn.email()`.
- Success → redirect `/`.
- Failure → show better-auth error (e.g. “Invalid credentials”).

**Implementation:** Client Component `SignInForm`; page is a thin Server Component wrapper (`app/auth/sign-in/page.tsx`).

### Sign up (`/auth/sign-up`)

**What they see**

- Same card style as sign-in.
- Name, email, password, confirm password; “Create Account”; errors; link to sign in.

**What they can do**

- Submit after client validation: passwords match, password length ≥ 8.

**On submit**

- Client validation first.
- Calls better-auth `signUp.email()`.
- Success → redirect `/` (session established).
- Failure → e.g. “Email already in use.”

**Implementation:** Client Component `SignUpForm`; page wrapper at `app/auth/sign-up/page.tsx`.

**Architecture note:** Middleware should send logged-in users away from `/auth/*` when appropriate.

## Add-to-favorites workflow

1. User clicks “Add to Favorites” in `FavoriteButton` (Client Component).

2. `FavoriteButton` calls `addFavoriteAction(productId)` (Server Action).

3. On the server, `addFavoriteAction`:
   - **a.** Calls `requireAuth()`:
     - better-auth checks session from cookies.
     - Session loaded via Prisma against Neon (e.g. session table lookup).
     - Valid session → `session.user.id`; invalid → redirect `/auth/sign-in`.
   - **b.** With `session.user.id`, `db.favorite.create({ data: { userId, productId } })` runs against Neon.
   - **c.** `revalidatePath('/favorites')` so the next visit to `/favorites` is fresh.

4. In the browser, optimistic UI may already show “Favorited ✓”; after success, visuals stay consistent.

5. User opens `/favorites` → Server Component loads favorites joined with products → grid shows the new item.
